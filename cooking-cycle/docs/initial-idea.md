# 📝 個人開発Webアプリ企画ドキュメント：献立ブレスト＆冷凍庫ストック "CookingSycle"

## 1. プロジェクト概要

### 1.1. 目的

本プロジェクトは、開発者自身の自炊における**「献立決定の複雑さ」**と**「冷凍庫の死蔵問題」**を解決するための実用的なWebアプリケーションを開発することを目的とする。学習を主眼とし、業務で得られない技術的な経験（DB設計、複雑な検索ロジック、モバイルUI/UX）を獲得する。

### 1.2. ユーザー（開発者自身）の課題（元々の発言）

* **レシピと献立決定の複雑性:**
    * 作るレシピはほぼ一緒だが、アレンジ（例：ペペロンチーノへのキャベツやしらすの追加）が派生し、管理が難しい。
    * 料理はYouTube動画を参考にしているため、作るたびに動画を探し、概要欄や動画内のレシピを確認するのが手間。
    * 結果として、考えるのが面倒になり、料理をしなくなる、または買い忘れる。
    * （発言引用：）「ペペロンチーノなんかほぼいつでも作れるんだけど、キャベツいれたりアンチョビ入れたりしらすいれたりするアレンジも派生して存在するし、管理難しいやん？」「スーパーでYouTubeの動画をイヤホンで聴いて食材探したりしてるし」
* **冷凍庫の死蔵問題:**
    * 作り置き（カレー、ミートソースなど）やまとめ買いした食材が冷凍庫に溜まり、何が残っているかわからない。
    * 結果、ダブり買いをしたり、せっかくのストックを献立決定時に活かせない。
    * （発言引用：）「冷凍庫死蔵問題があって」「そういうのがどんどん溜まる割に、今何が溜まってるのかわからないのよ」「だから、例えば鶏肉が冷凍してあるのにまたスーパーで買ってきちゃったりするし」

## 2. アプリケーションコンセプト

**「献立ブレスト＆冷凍庫ストック」**

**コア機能:** 柔軟なレシピ・アレンジ管理機能と、冷凍庫のストック状況を連動させた献立提案機能。

## 3. 主要機能要件 (Must-Have)

### 3.1. レシピ・アレンジ知識ベース機能

**機能の焦点:** レシピ情報の集約、柔軟な分類、および検索の実現。

| ID | 機能名 | 詳細要件 | 技術的要点 |
| :--- | :--- | :--- | :--- |
| **R-1** | 基本レシピ登録 | 基本となる料理（例：ペペロンチーノ）を登録。必須材料、基本手順、**元のYouTube URL**を紐づける。 | CRUD、リレーショナルDB設計 |
| **R-2** | アレンジレシピ登録 | 基本レシピの子として派生アレンジ（例：キャベツペペロンチーノ）を登録。**追加・変更材料**のみを記録し、手順は基本レシピを継承する。 | データの継承構造（親ID）、DBの親子リレーション |
| **R-3** | タグ付けシステム | レシピに対し、材料（鶏肉、パスタ）、味（さっぱり、スパイシー）、調理時間（15分以下）などの**自由なカスタムタグ**を設定する。 | **多対多リレーション** |
| **R-4** | YouTube連携 | 登録されたURLをクリックすると、アプリ内または外部で動画を再生し、**登録したタイムスタンプ**にジャンプできる。 | 外部URLの保存・リンク生成 |

### 3.2. 冷凍庫ストックメモ機能

**機能の焦点:** 在庫の把握と献立への活用。流動性の低い食材のみを対象とする。

| ID | 機能名 | 詳細要件 | 技術的要点 |
| :--- | :--- | :--- | :--- |
| **S-1** | ストック登録/更新 | **食材名**（鶏肉、ひき肉）または**作り置き名**（ミートソース、カレー）と、**量**（例：2食分、大）、**冷凍開始日**を登録・更新できる。 | モバイルでの高速入力UI、日付データ管理 |
| **S-2** | 場所メモ | 「上段奥の青いタッパー」など、冷凍庫内の**具体的な保存場所**をメモできるフィールドを設ける。 | シンプルなテキスト管理 |
| **S-3** | 消費推奨アラート | 冷凍開始日から設定した期間（例：1ヶ月）が経過した場合、**リスト上で目立つように表示する**。 | 日付計算ロジック、条件付き表示 |

### 3.3. 献立提案と連携機能

**機能の焦点:** 複雑な検索条件に対応し、ストック情報を活用したレシピを提案する。

| ID | 機能名 | 詳細要件 | 技術的要点 |
| :--- | :--- | :--- | :--- |
| **C-1** | 冷凍ストック優先フィルター | ワンクリックで、**現在ストックされている材料**を使って作れるレシピ（基本・アレンジ問わず）のみを一覧表示する。 | **レシピ材料タグとストックDBのマッチングロジック**、DBクエリ最適化（JOIN）|
| **C-2** | 高度な検索フィルター | 複数のタグ（R-3）を選択し、AND/OR/NOT（包含/除外）検索を実行できる。 | 複雑なクエリビルディング |
| **C-3** | 作り置きからの提案 | ストックされている作り置き（例：ミートソース）をタップすると、その作り置きを応用したアレンジレシピ（例：ドリア）を提案する。 | データの親子関係・逆引き参照 |

## 4. 技術的な学びのポイント (開発意図)

本アプリ開発を通じて、特に以下の技術領域のスキル向上を目指す。

* **データベース設計:** 複雑な**多対多リレーション**、および「基本レシピとアレンジ」の**継承構造**、「レシピとストック」の**連動構造**を持つDB設計。
* **検索アルゴリズム:** 複数の条件（タグ、ストック情報）に基づき、高性能かつ柔軟なフィルタリングを実現するクエリ最適化（SQL/NoSQL）。
* **フロントエンド/UI/UX:** モバイル環境（特にスーパーや台所）での操作性を考慮した**PWA（Progressive Web App）**の検討や、直感的なデータ入力フォームの設計。
* **外部API連携:** YouTube Data API（**将来的な拡張要素**として）の利用や、データキャッシュ戦略。
