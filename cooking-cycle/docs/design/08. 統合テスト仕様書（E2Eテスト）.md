# 統合テスト仕様書（E2Eテスト）

## 概要

E2Eテストのシナリオ、テストフロー、テスト環境構成を定義します。

## テスト環境

- **テストフレームワーク:** Playwright
- **ブラウザ:** Chromium, Firefox, WebKit
- **テスト環境:** Docker Compose（開発環境と同じ構成）

## E2Eテストシナリオ一覧

### 1. 認証フロー

#### 1.1 Google認証
**シナリオ:** ユーザーがGoogle認証でログインする

**テストフロー:**
1. ホーム画面にアクセス
2. 「ログイン」ボタンをクリック
3. Google認証ページにリダイレクト
4. Googleアカウントでログイン
5. ホーム画面にリダイレクト
6. ユーザー名が表示されることを確認

**期待結果:**
- 認証が成功する
- セッションが保持される
- ホーム画面が表示される

## テスト環境の構成

### Docker Compose構成

```yaml
version: '3.8'

services:
  app:
    build:
      context: .
      dockerfile: Dockerfile.test
    environment:
      - NODE_ENV=test
      - DATABASE_URL=postgresql://postgres:postgres@db:5432/cooking_cycle_test
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      - POSTGRES_USER=postgres
      - POSTGRES_PASSWORD=postgres
      - POSTGRES_DB=cooking_cycle_test
    volumes:
      - postgres_test_data:/var/lib/postgresql/data

volumes:
  postgres_test_data:
```

### テストデータの管理方法

#### セットアップ
- 各テストの前にテストデータをセットアップ
- テストデータは独立している（他のテストに影響しない）

#### クリーンアップ
- 各テストの後にテストデータをクリーンアップ
- データベースをリセット

#### テストデータファイル
- `tests/fixtures/users.json`: テストユーザー
- `tests/fixtures/recipes.json`: テストレシピ
- `tests/fixtures/stocks.json`: テストストック
- `tests/fixtures/products.json`: テスト製品

### テストファイルの例

```typescript
import { test, expect } from '@playwright/test';

test.describe('レシピ登録', () => {
  test.beforeEach(async ({ page }) => {
    // ログイン
    await page.goto('/');
    await page.click('text=ログイン');
    // Google認証のモック
  });

  test('レシピを登録できる', async ({ page }) => {
    // レシピ登録画面に遷移
    await page.click('text=レシピ登録');
    
    // フォームに入力
    await page.fill('input[name="name"]', 'ペペロンチーノ');
    await page.fill('textarea[name="instructions"]', '手順1\n手順2');
    
    // 登録ボタンをクリック
    await page.click('button[type="submit"]');
    
    // レシピ一覧画面に遷移
    await expect(page).toHaveURL('/recipes');
    
    // 登録したレシピが表示される
    await expect(page.locator('text=ペペロンチーノ')).toBeVisible();
  });
});
```

## Phase 2以降のE2Eテストシナリオ

### 状態遷移の整合性テスト（Phase 2以降）

#### シナリオ: 状態遷移に基づくレシピ検索

**目的**: ストックの状態遷移とレシピの要求状態が正しくマッチングされるか確認

**テストフロー:**
1. 食材マスタに「ひき肉」を登録
2. 処理マスタに「焼く」を登録
3. 状態遷移マスタに「ひき肉 + 焼く → そぼろ」を登録
4. ストックに「そぼろ（状態遷移ID指定）」を登録
5. レシピに「そぼろ（required_state_transition_id指定）」を材料として登録
6. ストック優先フィルターで検索
7. 期待結果: 該当レシピが表示される

### 在庫の整合性テスト（トランザクションテスト）

#### シナリオ: レシピ実行時の在庫消費と生成

**目的**: レシピを実行した際、材料となったストックの数量が正しく減算され、アウトプットのストックが生成されるか確認

**テストフロー:**
1. ストックA（材料1、2食分）を登録
2. ストックB（材料2、適量）を登録
3. レシピX（材料1、材料2 → 完成品1）を登録
4. レシピXを「実行」する（在庫消費処理）
5. 期待結果:
   - ストックAの数量が1食分に減る（または削除される）
   - ストックBが消費される（または数量が減る）
   - 完成品1のストックが生成される
   - トランザクションが正常に完了する

#### シナリオ: 在庫不足時のエラーハンドリング

**目的**: 在庫が不足している場合、適切にエラーを表示し、トランザクションがロールバックされるか確認

**テストフロー:**
1. ストックA（材料1、1食分）を登録
2. レシピX（材料1、2食分必要）を登録
3. レシピXを「実行」する
4. 期待結果:
   - 在庫不足エラーが表示される
   - トランザクションがロールバックされる
   - ストックAの数量が変更されない

### レシピの循環参照検出テスト（Phase 2以降）

#### シナリオ: 循環参照の検出とエラー表示

**目的**: レシピの依存関係で循環参照が発生した場合、適切にエラーを表示するか確認

**テストフロー:**
1. レシピA（材料: レシピBのアウトプット）を登録
2. レシピB（材料: レシピAのアウトプット）を登録しようとする
3. 期待結果:
   - 循環参照エラーが表示される
   - レシピBの登録が失敗する
