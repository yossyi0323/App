### **匿名投稿メモアプリ仕様書**

#### **1. 開発の前提と目的**
- **目的**: データベース、バックエンドAPI、フロントエンドUIを連携させた、一連のWebアプリケーション開発プロセスを体験すること。
- **機能**: ユーザー登録や認証は不要。誰でも匿名でテキストを投稿・閲覧できる単一のページアプリケーション。
- **技術スタック**: 実装を開始する前に、以下の技術要素を明示的に指定すること。
    - **フロントエンド**: [例: Vue.js, React, Svelte, Vanilla JavaScriptなど]
    - **バックエンド**: [例: Node.js (Express), Python (FastAPI), Go (Gin), Rust (Actix-web), Java (Spring Boot)など]
    - **データベース**: [例: PostgreSQL, MySQL, SQLite, DynamoDB, MongoDBなど]
    - **デプロイ環境**: [例: AWS (EC2/Lambda), Google Cloud Platform (Cloud Functions), Docker, Static Hosting Servicesなど]
- **コーディング支援**: AIコーディングエージェントの使用を前提とする。すべてのコード、テスト、ドキュメントは、エージェントが生成すること。

---

#### **2. 機能要件**

##### **2.1. フロントエンド (クライアント側)**
- **UIコンポーネント**: 単一のページ内に以下のコンポーネントを配置する。
    - **投稿フォーム**: 投稿内容を入力するためのテキストエリアと、投稿を実行するためのボタン（`Submit`）を持つ。
    - **投稿リスト**: 既存のすべての投稿を、新しい順にリスト形式で表示する。
- **動作**:
    - ページロード時に、既存のすべての投稿をバックエンドAPIから取得し、表示する。
    - 投稿フォームのボタンをクリックすると、テキストエリアの内容をバックエンドAPIに送信する。
    - 投稿が成功した場合、UIを再描画して、最新の投稿がリストに追加されていることを確認する。
    - 送信中のボタンの状態を「送信中...」のように変更するなど、ユーザーにフィードバックを提供すること。

##### **2.2. バックエンド (APIサーバー)**
- **APIエンドポイント**:
    - **POST /api/posts**: 投稿を新規作成するエンドポイント。
        - **リクエスト**: JSON形式で`{"content": "投稿内容"}`を受け取る。
        - **レスポンス**: 成功した場合、作成された投稿のIDを含むJSON（例：`{"id": 1}`）をHTTPステータスコード`201 Created`で返す。
    - **GET /api/posts**: すべての投稿を取得するエンドポイント。
        - **リクエスト**: なし。
        - **レスポンス**: すべての投稿を、新しい順に並べたJSON配列で返す（例：`[{"id": 1, "content": "内容", "createdAt": "日時"}]`）。
- **ロジック**:
    - データベースに接続し、データの読み書きを行うこと。
    - 投稿の作成時、`content`が空でないことを検証すること。

#### **3. 技術要件**

- **プロジェクト構成**:
    - `front`ディレクトリにフロントエンドのコードを配置する。
    - `server`ディレクトリにバックエンドのコードを配置する。
    - `database`ディレクトリにデータベースのスキーマファイル（SQLなど）を配置する。
- **データベーススキーマ**:
    - `posts`という名前のテーブルを作成する。
    - カラム: `id` (主キー, 自動採番), `content` (テキスト), `createdAt` (投稿日時, 自動生成)
- **テスト**:
    - **バックエンド**: APIエンドポイント（POST, GET）が正しく動作することを検証する統合テストを記述すること。
    - **フロントエンド**: 投稿フォームの動作や、投稿リストの表示を検証するコンポーネントテストを記述すること。
- **ドキュメンテーション**:
    - `README.md`ファイルに、プロジェクトのセットアップ方法、開発サーバーの起動方法、テストの実行方法を記述すること。

#### **4. デプロイ要件**

- **CI/CDパイプライン**:
    - GitLab CI/CD、GitHub Actions、または同等のツールを使用し、リポジトリへのプッシュ時に自動でテストを実行するパイプラインを構築すること。
- **本番環境**:
    - バックエンドとフロントエンドを、指定されたデプロイ環境にデプロイできる構成にすること。
    - 例: バックエンドはDockerコンテナとして、フロントエンドは静的ファイルとしてビルド・デプロイする。
