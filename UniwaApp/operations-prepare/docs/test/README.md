# テスト方針

## 1. テストの種類と構造

### 1.1 テストの種類
本プロジェクトでは、以下の3種類のテストを実装します：

1. **単体テスト（Unit Tests）**
   - 個々のコンポーネントやフックの動作を検証
   - 独立性が高く、他のコンポーネントに依存しない
   - 例：`InventoryItemCard`、`useInventoryStatus`など

2. **統合テスト（Integration Tests）**
   - 複数のコンポーネントが連携する機能を検証
   - コンポーネント間の相互作用を確認
   - 例：在庫確認画面全体の動作

### 1.2 ディレクトリ構造
```
operations-prepare/
├── __tests__/
│   ├── unit/                    # 単体テスト
│   │   ├── components/         # コンポーネントの単体テスト
│   │   │   └── inventory/
│   │   │       ├── inventory-item-card.test.tsx
│   │   │       └── place-selector.test.tsx
│   │   └── hooks/             # カスタムフックの単体テスト
│   │       └── use-inventory-status.test.ts
│   ├── integration/            # 統合テスト
│   │   └── inventory/
│   │       └── page.test.tsx   # ページ全体の統合テスト
```

## 2. テストの実装方針

### 2.1 単体テスト
- 各コンポーネントの独立性を保証
- 入力と出力の関係を明確に定義
- モックを使用して外部依存を排除
- エッジケースの考慮

### 2.2 統合テスト
- コンポーネント間の連携を確認
- データの流れを検証
- 状態管理の正確性を確認
- エラーハンドリングの検証

## 3. テストの実行

### 3.1 実行コマンド
```bash
# 単体テストの実行
npm run test:unit

# 統合テストの実行
npm run test:integration

# 全テストの実行
npm run test
```

### 3.2 テストの優先順位
1. 単体テスト（最も重要）
2. 統合テスト

## 4. テストの品質基準

### 4.1 カバレッジ
- 単体テスト: 100%
- 統合テスト: 100%

### 4.2 テストの品質チェック
- テストの独立性
- テストの再現性
- テストの保守性
- テストの実行速度

## 5. テストのメンテナンス

### 5.1 定期的なレビュー
- テストケースの見直し
- カバレッジの確認
- パフォーマンスの確認

### 5.2 ドキュメントの更新
- テスト仕様書の更新
- テストケースの追加・修正
- テスト結果の記録

## 6. 注意事項
- テストは常に最新の状態を維持
- テストの実行結果は記録を残す
- テストの失敗は即時対応
- テストコードも本番コードと同様にレビュー
